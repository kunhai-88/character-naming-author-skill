# 作者式角色取名

把姓名视为人物生活史的一部分，不把它当作性格标签或寓意谜语。

## 核心禁令

- 不从“坚韧、清醒、深沉、温柔”等人物标签直接拼字。
- 不先生成二十个名字让用户抽奖。
- 不以字面寓意漂亮作为主要推荐理由。
- 不默认使用现成网文姓氏与高频气质字。
- 不把“读起来像主角”当作优点；现实题材尤其警惕姓名抢戏。
- 不因改主角名而破坏亲属姓氏、称呼、年代或地域逻辑。

## 工作流程

### 1. 先读故事，再看名字

优先读取完整剧本；材料过长时至少读取：人物表、开场、主角第一次受压、一次亲密关系场面、一次公开冲突和结尾。

提取：

- 故事年份、地域、城乡与阶层；
- 人物出生年份而非故事发生年份；
- 家庭成员、姓氏来源和称呼方式；
- 谁拥有取名权；
- 主角在对白里被怎样叫；
- 同剧高频人物名及其声音距离。

缺少材料时，只追问会实质改变姓名的事实。若能从剧本推断，标明假设并继续。

### 2. 建立“取名现场”

在想候选前，先用一小段内部推理回答：

> 这个名字由谁、在什么年份、什么家庭处境下取出？取名者当时知道什么，又不知道人物未来会成为什么？

名字可以承载父母的愿望、排行、宗族、时代风气、登记习惯或偶然，但不能精准预言剧情。

需要详细判断时读取 [references/author-naming-method.md](references/author-naming-method.md)。

### 3. 先定姓名带宽

不要直接定具体名字。先界定这个人物真实可能出现的范围：

- 常见或少见；
- 单名或双名；
- 朴素、书卷、地方、宗族、单位大院或新中产来源；
- 本名、乳名、外号、艺名是否并存；
- 姓氏是否由父系、母系、继亲、收养或族群规则决定。

### 4. 生成少量异源候选

生成 4—8 个内部候选，但它们必须来自不同的取名原因，而非同一批字的排列组合。例如：

- 家庭真实惯例；
- 年代常见取法；
- 地域口语与乳名；
- 与人物命运形成不刻意的反差；
- 极普通但能被角色“演成专有名”的名字。

默认只向用户交付 1 个首选和最多 2 个备选。

### 5. 放回剧本试演

至少完成四种试读：

1. 长辈日常叫他；
2. 亲密关系低声叫他；
3. 对手在冲突中喊全名；
4. 陌生人、老师、医生、警察或同事正式称呼他。

检查名字是否需要作者解释才能成立，是否在激烈对白中含混，昵称是否自然。

### 6. 做盲审，不替候选辩护

隐藏取名寓意，只把姓名、人物年龄、关系和对白片段交给“编辑视角”判断：

- 像父母取的，还是像编剧取的？
- 像真实人物，还是像类型标签？
- 姓名是否比人物先表演？
- 换到十部同类短剧里是否仍毫无违和？若是，可能过于模板化。
- 与同剧人物是否一耳朵能分开？

读取 [references/anti-ai-audit.md](references/anti-ai-audit.md) 执行审查。若盲审不通过，回到取名现场重做，不靠解释挽救。

### 7. 交付明确结论

按以下顺序输出：

1. **首选姓名**；
2. **为什么属于这个家庭和年代**；
3. **放回台词的三至四句试演**；
4. **原名保留或更换结论**；
5. 必要时给最多两个备选，并说明它们改变了什么人物感觉。

若审查整部剧，先给总体姓名系统结论，再逐个列出“保留 / 建议更换 / 必须更换”。不要为了展示工作量而把所有名字都改掉。

## 确定性检查

需要检查同剧姓名重复、共享字、疑似模板字时运行：

```bash
python3 scripts/name_audit.py 姓名1 姓名2 姓名3
```

脚本只负责提醒，不负责文学判断。不得把命中词表直接当作判死刑。

## 成功标准

一个好名字应当同时通过三关：

- **出生关**：人物的父母真可能在那个时间地点取出它；
- **开口关**：不同关系的人都能自然叫出口；
- **遗忘关**：作者解释被删掉后，名字仍然成立，人物演完后观众才觉得它只能属于他。
